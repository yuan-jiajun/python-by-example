# py-explain：Python 源码中文精讲

把指定的 Python 文件（或目录）做**重点 + 难点级**的语法解析、机制剖析和避坑分析，
并将中文注释**直接写进源码**。目标是把每个 .py 文件变成一份可以边读边学的中文讲义。

本项目（python-by-example）的文件已有大量**英文注释**，本 skill 在其基础上叠加中文，
而不是推翻重写。

## 核心约束（务必遵守）

1. **语言**：所有新增注释一律用**中文**。
2. **保留英文**：原有英文注释、docstring 一律**保留不动**，中文写在它**下方/旁边**作为补充。
3. **只加注释，不改逻辑**：绝不修改任何可执行代码、变量名、缩进结构。唯一的改动是增删注释行。
4. **深度 = 重点 + 难点级**：按「代码块 / 关键行」讲解，**不要逐行**翻译。
   把笔墨花在：语法机制为什么这样、初学者容易踩的坑、Python 的惯用法 / 最佳实践、
   与其他语言的差异。显而易见的行（如 `print(x)`）不必注释。
5. **避坑双层落地**：
   - **就近内联**：在相关代码旁用 `# ⚠️ 坑：...` 标出易错点。
   - **文件尾汇总**：在文件末尾追加一个「避坑总结」区块统一回顾（见模板）。
6. **改完必须验证**：每个文件注释完成后，用 `python <file>` 实际跑一次，
   确认没有因为加注释而破坏代码 / 缩进 / 语法。跑挂了要立刻修回来。

## 工作流程

### 输入是单个文件
1. 用 Read 完整读取该文件，理解它要教的主题和每个分区。
2. 按下方「注释风格」就地补充中文注释。
3. 在文件末尾追加「避坑总结」区块。
4. 用 `python <file>` 运行验证；有报错就修复后重跑，直到通过。
5. 向用户简要汇报：讲了哪些重点、标了哪些坑、运行是否通过。

### 输入是目录
1. 先用一次命令递归列出目录下所有 `.py` 文件：
   ```bash
   find <dir> -type f -name "*.py" | sort
   ```
2. **先做全局概览分析**：依次 Read 每个文件，提炼出**每个文件的侧重点**
   （它主要教什么、核心难点是什么），形成一份「概览清单」展示给用户。例如：
   ```
   目录 09_advanced_python/ 概览（共 4 个文件）：
   [1] 01_list_comprehensions.py — 列表/字典/集合推导式；侧重点：嵌套推导、条件过滤、可读性边界
   [2] 02_generators.py          — 生成器与 yield；侧重点：惰性求值、内存优势、send/close
   [3] 03_decorators.py          — 装饰器；侧重点：闭包、@wraps、带参装饰器三层嵌套
   [4] 04_context_managers.py    — 上下文管理器；侧重点：with 协议、__enter__/__exit__、contextlib
   ```
   这一步只读不写，目的是先摸清全貌、确定每个文件讲解的重心，再开工。
3. 展示概览清单后等用户确认（或让用户挑选要先做哪些）。
4. 用户确认后，**按清单顺序逐个处理**，每个文件走完整的「单文件流程」，
   并依据概览阶段定下的侧重点来组织讲解重心。
5. 每完成一个文件，简要汇报进度（如 `[2/5] 已完成 03_decorators.py，验证通过`），
   再继续下一个。

## 注释风格

参考本项目现有风格（`"""文件头 docstring"""` + `# ---- 分区标题 ----`），中文与之协调。

### 1. 文件头：在原英文 docstring 之后补一段中文导读
不要删原 docstring，在它**下方**新增一个中文导读注释块：

```python
"""
================================================================================
File: 03_decorators.py
... (原有英文 docstring 原样保留) ...
================================================================================
"""

# ============================================================================
# 【中文导读】装饰器（Decorator）
# ----------------------------------------------------------------------------
# 一句话本质：装饰器就是「一个接收函数、返回新函数」的函数，用 @ 语法糖套在
# 被装饰函数头上，从而在不改动原函数代码的前提下增强它（日志/计时/鉴权/缓存…）。
#
# 学这个文件你要拿下的难点：
#   1. 为什么 @decorator 等价于 func = decorator(func)
#   2. 闭包：wrapper 为什么能记住外层的 func
#   3. *args / **kwargs 如何让 wrapper 适配任意签名
#   4. @wraps 解决了什么（否则元数据被 wrapper 覆盖）
#   5. 带参数装饰器为什么要套三层
# ============================================================================
```

### 2. 代码块/关键行：就近补中文
在分区标题下、或关键语句旁，用中文讲清「机制」和「为什么」：

```python
def my_decorator(func):
    def wrapper(*args, **kwargs):   # 中文：wrapper 用 *args/**kwargs 接收任意参数，
                                    #       这样能装饰任何签名的函数，再原样转发给 func
        print("  Before function call")
        result = func(*args, **kwargs)
        return result
    return wrapper                  # 中文：返回的是函数对象本身，注意没有 ()，不是调用
```

### 3. 易错点：就近用 ⚠️ 标记

```python
@my_decorator
def say_hello_v2(name):
    ...
# ⚠️ 坑：不加 @wraps 时，say_hello_v2.__name__ 会变成 'wrapper'，
#        调试和文档工具会显示错误的函数名。
```

### 4. 文件末尾：避坑总结区块
在文件最后追加（放在原有 Summary 之后即可，不删原 Summary）：

```python
# ============================================================================
# 【避坑总结 · 03_decorators.py】
# ----------------------------------------------------------------------------
# 1. 闭包延迟绑定：装饰器里如果在循环中创建 wrapper，注意变量捕获的是引用不是值。
# 2. 忘记 return wrapper → 被装饰函数变成 None。
# 3. 不用 @functools.wraps → 丢失 __name__/__doc__，影响内省与调试。
# 4. 带参数装饰器必须三层嵌套：repeat(times) → decorator(func) → wrapper(*a,**kw)。
# 5. 多个装饰器自下而上应用：@bold @italic f 等价于 bold(italic(f))。
# ============================================================================
```

## 输出汇报格式

每个文件处理完后，向用户用中文汇报：
- ✅ 已完成的文件名
- 📌 讲解的核心难点（3-5 条）
- ⚠️ 标记的主要坑（若干条）
- ▶️ 运行验证结果（通过 / 报错及处理）

## 注意事项
- 注释要有**你自己的理解和思考**，不是机械翻译英文，可补充原文没讲到的机制、
  对比、惯用法。
- 行内注释不要太长，超长的解释放成独立的 `#` 注释行块。
- 若某文件已经被本 skill 处理过（已有【中文导读】/【避坑总结】），先告知用户，
  询问是覆盖更新还是跳过。
