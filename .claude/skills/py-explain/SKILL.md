# py-explain：Python 源码中文精讲

把指定的 Python 文件（或目录）做**重点 + 难点级**的语法解析、机制剖析和避坑分析，
并将中文注释**直接写进源码**。目标是把每个 .py 文件变成一份可以边读边学的中文讲义。

本项目（python-by-example）的文件原本是**英文注释**，本 skill 会把这些英文注释
**改写成中文**（替换，不保留英文），并在此基础上补充更深入的中文讲解。

**读者画像**：使用者是有经验的 **Java 开发者**，正在转学 Python。因此讲解要
**主动结合 Java 做对比**——指出同一概念在两门语言里的写法/行为差异、Java 里没有
而 Python 特有的东西、以及 Java 习惯带过来会踩的坑。

## 核心约束（务必遵守）

1. **语言**：所有注释一律用**中文**，**不要留英文**。
2. **英文融进中文**：遇到原有英文注释 / docstring，**不要单独翻译成一行、再另起一块写讲解**，
   而是把原注释的意思**融合进你的中文讲解里**，合成**一条连贯的中文注释**。
   即：最终不出现“翻译行 + 讲解块”两段并列，只有一段融合后的中文。文件头 docstring 同理改写。
3. **结合 Java 对比**：在合适的地方用 `# Java 对比：...` 点出与 Java 的差异，例如
   类型系统、`==` 与 equals、检查型异常、`final`/常量、自动装箱、`null` vs `None`、
   重载、接口、泛型、字符串不可变性等。对比要准确、点到为止，不喧宾夺主。
4. **只加注释，不改逻辑**：绝不修改任何可执行代码、变量名、缩进结构。唯一的改动是注释。
   （例外：若原代码本身有 bug 导致无法运行，做最小修复并用 `# ⚠️` 注明，并向用户报告。）
5. **深度 = 重点 + 难点级**：按「代码块 / 关键行」讲解，**不要逐行**翻译。
   把笔墨花在：语法机制为什么这样、Java 转 Python 易踩的坑、Python 的惯用法 / 最佳实践。
   显而易见的行（如 `print(x)`）不必注释。
6. **避坑双层落地**：
   - **就近内联**：在相关代码旁用 `# ⚠️ 坑：...` 标出易错点。
   - **文件尾汇总**：在文件末尾追加一个「避坑总结」区块统一回顾（见模板）。
7. **改完必须验证**：每个文件注释完成后，用 `python3 <file>`（本机无 `python`，统一用 `python3`） 实际跑一次，
   确认没有因为加注释而破坏代码 / 缩进 / 语法。跑挂了要立刻修回来。

## 工作流程

### 输入是单个文件
1. 用 Read 完整读取该文件，理解它要教的主题和每个分区。
2. 按下方「注释风格」就地补充中文注释。
3. 在文件末尾追加「避坑总结」区块。
4. 用 `python3 <file>`（本机无 `python`，统一用 `python3`） 运行验证；有报错就修复后重跑，直到通过。
5. 向用户简要汇报：讲了哪些重点、标了哪些坑、运行是否通过。

### 输入是目录
1. 先用一次命令递归列出目录下所有 `.py` 文件：
   ```bash
   find <dir> -type f -name "*.py" | sort
   ```
2. **先做全局概览分析**：依次 Read 每个文件，提炼出**每个文件的侧重点**
   （它主要教什么、核心难点是什么），形成一份「概览清单」展示给用户。例如：
   ```
   目录 09_advanced_python/ 概览（共 4 个文件）：
   [1] 01_list_comprehensions.py — 列表/字典/集合推导式；侧重点：嵌套推导、条件过滤、可读性边界
   [2] 02_generators.py          — 生成器与 yield；侧重点：惰性求值、内存优势、send/close
   [3] 03_decorators.py          — 装饰器；侧重点：闭包、@wraps、带参装饰器三层嵌套
   [4] 04_context_managers.py    — 上下文管理器；侧重点：with 协议、__enter__/__exit__、contextlib
   ```
   这一步只读不写，目的是先摸清全貌、确定每个文件讲解的重心，再开工。
3. 展示概览清单后等用户确认（或让用户挑选要先做哪些）。
4. 用户确认后，**按清单顺序逐个处理**，每个文件走完整的「单文件流程」，
   并依据概览阶段定下的侧重点来组织讲解重心。
5. 每完成一个文件，简要汇报进度（如 `[2/5] 已完成 03_decorators.py，验证通过`），
   再继续下一个。

## 注释风格

参考本项目现有风格（`"""文件头 docstring"""` + `# ---- 分区标题 ----`），中文与之协调。

### 1. 文件头：把英文 docstring 改写成中文导读（合并进同一个 docstring）
将顶部英文 docstring **整体改写成中文**（保留原有 `===` 边框风格），把“导读 / 本质 /
难点 / Java 对比”**全部写进这同一个 docstring 里**。
**不要**在 docstring 下方再单独加一个 `# ===【中文导读】===` 注释块——文件头只保留这一处。

```python
"""
================================================================================
文件：03_decorators.py
主题：Python 装饰器（Decorator）
================================================================================

一句话本质：装饰器就是「一个接收函数、返回新函数」的函数，用 @ 语法糖套在
被装饰函数头上，在不改动原函数代码的前提下增强它（日志/计时/鉴权/缓存…）。

Java 对比：类似 Java 的注解(@Override 等) + AOP 切面，但更直接——Python 装饰器
就是普通函数，运行期真实地把函数包了一层，不需要反射/字节码增强框架。

学这个文件你要拿下的难点：
  1. 为什么 @decorator 等价于 func = decorator(func)
  2. 闭包：wrapper 为什么能记住外层的 func
  3. *args / **kwargs 如何让 wrapper 适配任意签名
  4. @wraps 解决了什么（否则元数据被 wrapper 覆盖）
  5. 带参数装饰器为什么要套三层
================================================================================
"""
```

### 2. 代码块/关键行：用中文讲清机制，并随手做 Java 对比
把原英文行内注释、以及分区标题下那句英文小注（如 `# print() can output any data type`），
**融进你的中文讲解里写成一条连贯注释**，不要保留成单独的翻译行。
在关键语句旁讲清「机制」和「为什么」，合适处加 `# Java 对比：...`：

```python
def my_decorator(func):
    def wrapper(*args, **kwargs):   # wrapper 用 *args/**kwargs 接收任意参数，
                                    # 这样能装饰任何签名的函数，再原样转发给 func
        print("  调用前")
        result = func(*args, **kwargs)
        return result
    return wrapper                  # 返回函数对象本身，没有 ()，不是调用
# Java 对比：函数在 Python 里是「一等对象」，可直接当值传递/返回；
#            Java 8 前只能靠接口/匿名类，8 之后才有 Lambda 和方法引用近似。
```

### 3. 易错点：就近用 ⚠️ 标记

```python
@my_decorator
def say_hello_v2(name):
    ...
# ⚠️ 坑：不加 @wraps 时，say_hello_v2.__name__ 会变成 'wrapper'，
#        调试和文档工具会显示错误的函数名。
```

### 4. 文件末尾：避坑总结区块
在文件最后追加（原有 Summary 若是英文，一并改写成中文）：

```python
# ============================================================================
# 【避坑总结 · 03_decorators.py】
# ----------------------------------------------------------------------------
# 1. 闭包延迟绑定：装饰器里如果在循环中创建 wrapper，注意变量捕获的是引用不是值。
# 2. 忘记 return wrapper → 被装饰函数变成 None。
# 3. 不用 @functools.wraps → 丢失 __name__/__doc__，影响内省与调试。
# 4. 带参数装饰器必须三层嵌套：repeat(times) → decorator(func) → wrapper(*a,**kw)。
# 5. 多个装饰器自下而上应用：@bold @italic f 等价于 bold(italic(f))。
# ============================================================================
```

## 输出汇报格式

每个文件处理完后，向用户用中文汇报：
- ✅ 已完成的文件名
- 📌 讲解的核心难点（3-5 条）
- ⚠️ 标记的主要坑（若干条）
- ▶️ 运行验证结果（通过 / 报错及处理）

## 注意事项
- 注释要有**你自己的理解和思考**，不是机械翻译英文，可补充原文没讲到的机制、
  对比、惯用法。
- **结合 Java 对比**是核心要求：用户是 Java 老手，凡是两门语言差异明显、或 Java 习惯
  会导致踩坑的地方，都应主动点出。但对比要准确、克制，不为对比而对比。
- 全文不留英文注释；原英文 docstring / 行内注释 / Summary 一律改写为中文。
- 行内注释不要太长，超长的解释放成独立的 `#` 注释行块。
- 若某文件已经被本 skill 处理过（已有【中文导读】/【避坑总结】），先告知用户，
  询问是覆盖更新还是跳过。
